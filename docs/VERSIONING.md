# Версионна Система на KCY Ecosystem

## Два вида версии

### 1. Файл-версия (името на файла)
**Файл:** `XXXXX.version` в главната директория (напр. `00033.version`)

**Числото в името се сменя при почти всяка промяна на код.** По него се ориентираме дали има нов код в доставката. Това е брояч на доставки/билдове.

Пример: `00032.version` → `00033.version` → `00034.version`

### 2. Вътрешна версия (съдържанието)
**Съдържанието** (напр. `1.0085`) се сменя по правилата за значими промени:

- ✅ Променена логика или архитектура
- ✅ Добавени нови функции или модули
- ✅ Поправени значими бъгове
- ✅ Променена структура на проекта
- ❌ Малки коментари, форматиране, козметика

### Кога какво се сменя?

| Промяна | Файл-версия (име) | Вътрешна версия |
|---------|-------------------|-----------------|
| Нова функция | ✅ +1 | ✅ bump |
| Bug fix в логика | ✅ +1 | ✅ bump |
| Промяна в архитектура | ✅ +1 | ✅ bump |
| Нов deploy скрипт | ✅ +1 | ✅ bump |
| Добавяне на тестове | ✅ +1 | по преценка |
| Само документация | ✅ +1 | ❌ |
| Козметична промяна | ❌ | ❌ |

## Версии на отделни файлове

Всеки файл може да има СВОЯ версия, независимо от другите.

### Формат по тип файл

**JavaScript:**
```javascript
// Version: 1.0085
```

**Solidity:**
```solidity
/**
 * @version v34
 */
```

**Markdown:**
```markdown
<!-- @version v34 -->
```

**Bash скриптове:**
```bash
# Version: 1.0085
```

### Правила
1. Версията е **НАЙ-ОТГОРЕ** във файла
2. Версията е само на **ЕДНО място**
3. Всеки файл има **СВОЯ версия**
4. Увеличавай при **значителни промени**
5. НЕ увеличавай за **малки коментари**

## Проверка на версии

```bash
# Виж текущата версия на проекта
cat *.version

# JavaScript файлове
grep -r "Version:" --include="*.js" -l | head -20

# Solidity файлове
grep -r "@version" --include="*.sol"

# Bash скриптове
grep -r "Version:" --include="*.sh"
```
