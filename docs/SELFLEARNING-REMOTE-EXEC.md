# Selflearning Friend — изпълнение на команди на други машини (SSH)

Роботчето може да се свърже с друга машина и да изпълни команди там — резултатът
се връща в чата. Това ръководство обяснява архитектурата, сигурността, как се
ВКЛЮЧВА на сървъра и как се ползва от приложението.

---

## Архитектура (защо е през relay)

Приложението живее в Android WebView и **няма достъп до суров TCP сокет** → телефонът
**не може сам да прави SSH**. Затова:

```
телефон → HTTPS → наш relay (kcy-selflearning) → SSH → целевата машина → изход обратно в чата
```

- Телефонът праща само `{ host, command }` към `POST /api/selflearning/exec/<token>`.
- Relay-ят (който държи SSH достъпа) изпълнява и връща `{ stdout, stderr, code }`.
- **Телефонът НЕ пази SSH ключове** — те стоят на сървъра.
- Ако `host` е празен / `localhost` → командата се изпълнява **локално на relay сървъра**.

---

## Сигурност (важно)

| Слой | Защита |
|------|--------|
| **Изключено по подразбиране** | Endpoint-ът работи САМО ако `SELFLEARNING_EXEC_ENABLED=1` на сървъра. Без това → `403 exec_disabled`. Relay-ят не е „тихо" RCE кутия. |
| **Кодова дума** | Командата в апа е гейтната: трябва `„<кодова дума>, …!"`. Без вярната дума апът дори не праща заявка. |
| **Token** | Стандартният token намespace + rate-limit важат. |
| **Лог** | Всяка команда се записва в журнала на услугата (`journalctl -u kcy-selflearning`). |
| **Таймаут + таван** | По подразбиране 60 сек и ~100 KB изход на поток. |
| **Достъп на relay-а** | SSH стига дотам, докъдето стига ключът на relay-а — не по-далеч. |

> ⚠ Това е **пълен шел** на машините, до които relay-ят има SSH достъп. Token-ът става
> мощен credential — пази го. Включвай `SELFLEARNING_EXEC_ENABLED` само когато ти трябва.

## Как се ВКЛЮЧВА на сървъра

1. В `private/configs/.env` (чете се от услугата като `EnvironmentFile`) добави:

   ```bash
   SELFLEARNING_EXEC_ENABLED=1
   # по избор — SSH параметри, с които relay-ят се връзва към ЦЕЛЕВИТЕ машини:
   SELFLEARNING_EXEC_SSH_USER=deploy          # потребител (по подразбиране deploy)
   SELFLEARNING_EXEC_SSH_PORT=2222            # порт (по подразбиране 22)
   SELFLEARNING_EXEC_SSH_KEY=/home/deploy/.ssh/id_ed25519   # по избор: частен ключ
   SELFLEARNING_EXEC_TIMEOUT_MS=60000         # таймаут на команда
   SELFLEARNING_EXEC_MAX_OUTPUT=100000        # таван на изхода (байта/поток)
   ```

2. Увери се, че relay-ят има SSH достъп до целевите машини (ключът му е в техния
   `authorized_keys`). За команди НА самия сървър (localhost) това не е нужно.

3. Рестартирай услугата (опция 38, или ръчно):
   ```bash
   sudo systemctl restart kcy-selflearning
   ```

4. Проверка: `curl -s https://<домейн>/api/selflearning/health` трябва да върне `ok`.

## Как се ползва от приложението

Всичко е гейтнато с **кодовата дума** (името на робота). Формат: `„<дума>, …!"`.

| Казваш | Какво прави |
|--------|-------------|
| `Пешо, свържи се с take.offbitch.com!` | Свързва се и прави проба (`hostname/whoami/uptime`); помни хоста. |
| `Пешо, свържи се с take.offbitch.com и изпълни df -h!` | Свързва се и изпълнява `df -h` веднага. |
| `Пешо, изпълни на take.offbitch.com systemctl status nginx!` | Изпълнява на конкретната машина. |
| `Пешо, изпълни uptime!` | Изпълнява на **последно свързаната** машина. |
| `Пешо, ssh до 192.168.0.108 и изпълни whoami!` | Алтернативен изказ. |
| `Пешо, изпълни на сървъра ls -la!` | „сървъра/relay/локално" → на самия relay сървър. |

Изходът се връща в чата (stdout + stderr + кодът на изход). Без вярната кодова дума
апът отговаря с дружелюбен отказ и НЕ праща нищо.

---

## Честни ограничения

- Целевата машина трябва да е достъпна за relay-а по SSH (ключ/мрежа). Ако не е —
  връща се SSH грешката (напр. „Permission denied", „Connection timed out").
- Това НЕ е интерактивна сесия — всяка команда е отделно изпълнение (без запазено
  състояние между команди, освен текущата работна машина, която апът помни).
- Гласово диктуване на сложни команди (със символи) е ненадеждно — за команди пиши.
